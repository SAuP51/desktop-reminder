下面先按“**极低资源占用、Windows 桌面端、后台运行、屏幕滚动飘字提醒**”来明确需求与方案，不写代码。

## 1. 需求边界先定清楚

这个产品实际应拆成两类能力：

**A. 后台调度能力**

负责保存提醒、计算下一次触发时间、低功耗等待、错过补偿、系统唤醒/恢复后的重算。

**B. 用户会话内的显示能力**

负责在当前用户桌面上创建一个透明、置顶、可选穿透鼠标的悬浮窗口，让提醒文字从屏幕一侧滚动飘过。

这里有一个重要架构点：**不建议把整个程序做成传统 Windows Service 并直接画 UI**。Windows 服务运行在 Session 0，而从 Windows Vista 开始 Session 0 专门保留给服务和非交互程序，不支持与用户交互；微软文档也说明服务默认使用非交互窗口站，交互式服务不应在高权限上下文里创建桌面窗口。([微软学习](https://learn.microsoft.com/en-us/windows/win32/services/service-changes-for-windows-vista?utm_source=chatgpt.com))

所以推荐架构是：

**主方案：单用户后台 Agent**

用户登录后自动启动一个 Rust 桌面后台进程，例如 `reminder-agent.exe`。它平时不显示窗口，只保留极少线程和一个等待中的系统定时器。需要提醒时才创建/显示悬浮层。

**增强方案：Service + Agent 双进程**

如果以后需要“开机未登录也记录状态”“多用户统一管理”“企业部署”“自动更新”等，再加一个无 UI 的 Windows Service。Service 只做后台协调，不直接画 UI；真正飘字仍由每个登录用户会话里的 Agent 完成。

自动启动建议用 Windows Task Scheduler 的登录触发器，而不是塞进启动文件夹。微软的 LogonTrigger 表示“用户登录时启动任务”，Task Scheduler 服务启动时也会枚举已登录用户并运行匹配的登录触发任务。([微软学习](https://learn.microsoft.com/zh-cn/windows/win32/taskschd/logontrigger?utm_source=chatgpt.com))

## 2. 技术选型建议

### 语言与系统 API

核心用 **Rust + Win32 API**。Rust 侧建议通过 Microsoft 的 `windows` crate 调用 Windows API；微软文档说明 Rust for Windows 可以通过 `windows` crate 直接使用 Windows API。([微软学习](https://learn.microsoft.com/en-us/windows/dev-environment/rust/rust-for-windows?utm_source=chatgpt.com))

不建议 Electron/Tauri 作为第一版核心实现，因为它们更适合快速做管理界面，不适合“常驻、极低内存、极低唤醒频率、只在提醒时绘制”的目标。

### 定时机制

使用 **Waitable Timer + 最小堆调度队列**。

Windows 的 `SetWaitableTimer` 支持一次性或周期性定时器，到期后定时器对象会变成 signaled；其 `lPeriod` 可设置周期，`0` 表示一次性定时器。它还支持 `fResume` 参数，但如果系统不支持唤醒会返回 `ERROR_NOT_SUPPORTED`。([微软学习](https://learn.microsoft.com/zh-tw/windows/win32/api/synchapi/nf-synchapi-setwaitabletimer?utm_source=chatgpt.com))

这里不要给每个提醒都创建一个系统定时器。应该只维护一个全局系统定时器，永远指向“最近的一次提醒”。这样提醒数量再多，后台也只需要等待一个内核对象。

### 悬浮层渲染

悬浮层用原生 Win32 窗口，样式可考虑：

`WS_EX_LAYERED`：透明/半透明窗口。
`WS_EX_TOPMOST`：始终置顶。
`WS_EX_TOOLWINDOW`：不在任务栏显示。
`WS_EX_TRANSPARENT`：可选，用于鼠标点击穿透。

这些扩展窗口样式可以与 `CreateWindowEx` 配合使用。([微软学习](https://learn.microsoft.com/zh-cn/windows/win32/winmsg/extended-window-styles?utm_source=chatgpt.com))

文字渲染建议两档：

第一版：**Direct2D + DirectWrite**。Direct2D 是硬件加速的 2D 图形 API，适合高性能渲染几何、位图和文本。([微软学习](https://learn.microsoft.com/en-us/windows/win32/direct2d/direct2d-portal?utm_source=chatgpt.com))
进阶版：**DirectComposition** 托管动画。DirectComposition 支持高性能位图合成、变换和动画，并可使用图形硬件、独立于 UI 线程执行流畅转换。([微软学习](https://learn.microsoft.com/zh-cn/windows/win32/directcomp/directcomposition-portal?utm_source=chatgpt.com))

第一版不需要一上来做 DirectComposition。用普通 Win32 消息循环 + Direct2D 每帧绘制，控制好动画帧率即可。

### 存储

本地存储建议 SQLite。它可靠、部署简单、适合小型本地日程数据库。若启用 WAL，SQLite 文档说明读操作可以和写入 WAL 的提交记录并行存在，读者会基于自己的 end mark 读取一致视图。([SQLite](https://sqlite.org/wal.html?utm_source=chatgpt.com))

第一版数据量很小，也可以先用 JSON/TOML，但一旦涉及提醒历史、规则版本、同步、迁移，SQLite 会更稳。

## 3. 推荐进程模型

### 第一版最小可行架构

```text
reminder-agent.exe
 ├─ Scheduler Thread        计算下一次提醒 + 等待系统定时器
 ├─ UI Thread               Win32 消息循环，平时隐藏窗口
 ├─ Storage Layer           SQLite / 配置文件
 ├─ Rule Engine             解析日程规则，计算 next_fire_at
 ├─ Overlay Renderer        透明置顶窗口 + 文字滚动动画
 └─ IPC Server 可选          给设置界面/CLI 修改提醒
```

更极致的低资源模式：

```text
reminder-agent.exe
 ├─ 主线程：消息循环 + 隐藏窗口 + 系统事件监听
 └─ 调度线程：WaitableTimer 阻塞等待
```

平时 CPU 使用应接近 0，因为它不是轮询，而是在等待最近一次定时器触发。

## 4. 日程规则模型

不要只设计成“某天某时提醒”。你的需求里有“时段”和“固定时间间隔”，所以规则层要比普通闹钟更通用。

建议内部模型：

```text
Reminder
 ├─ id
 ├─ title
 ├─ message
 ├─ enabled
 ├─ timezone
 ├─ priority
 ├─ display_policy
 │   ├─ duration_seconds
 │   ├─ speed_px_per_sec
 │   ├─ position: top / middle / bottom / custom
 │   ├─ repeat_on_screen: once / n_times / until_ack
 │   └─ click_through: true / false
 └─ schedule_rule
     ├─ date_range: start_date ~ end_date?
     ├─ day_filter: weekdays / monthdays / specific_dates / rrule
     ├─ time_windows
     │   ├─ 09:00~18:00 every 25 minutes
     │   ├─ 22:00~23:00 every 5 minutes
     │   └─ fixed_times: [10:00, 14:30, 20:00]
     ├─ exclusions
     └─ missed_policy: skip / fire_once / fire_all_limited
```

例子：

**每天 9:00 到 18:00，每 30 分钟提醒一次**

```text
date_range: unlimited
day_filter: every_day
time_windows:
  - start: 09:00
    end: 18:00
    interval: 30m
```

**工作日 10:00、14:00、17:30 各提醒一次**

```text
day_filter: Mon-Fri
time_windows:
  - fixed_times: [10:00, 14:00, 17:30]
```

**某日期范围内，每隔 2 小时提醒一次**

```text
date_range: 2026-07-01 ~ 2026-07-31
time_windows:
  - start: 00:00
    end: 23:59
    interval: 2h
```

## 5. 核心调度算法

核心思想：**所有提醒都预先计算下一次触发时间，放入最小堆。系统只等待堆顶。**

```text
启动时：
  1. 从存储加载所有 enabled reminders
  2. 对每条规则计算 next_fire_at
  3. 放入 MinHeap，key = next_fire_at
  4. 设置 WaitableTimer 到最早的 next_fire_at

定时器触发：
  1. 弹出所有 next_fire_at <= now 的提醒
  2. 发送给 Overlay Renderer 显示
  3. 对这些提醒重新计算下一次 next_fire_at
  4. 放回 MinHeap
  5. 重设 WaitableTimer 到新的堆顶
```

### 为什么不用每秒扫描？

每秒扫描很简单，但不符合“极低资源占用”。最小堆 + 单个系统定时器的复杂度更好：

```text
新增/修改提醒：O(log n)
触发提醒：O(log n)
查找最近提醒：O(1)
后台空闲 CPU：接近 0
```

## 6. “时段 + 间隔”的计算方法

对于规则：

```text
窗口：09:00 ~ 18:00
间隔：25 分钟
锚点：09:00
当前时间：10:13
```

计算方式：

```text
elapsed = now_time - window_start
k = ceil(elapsed / interval)
candidate = window_start + k * interval
```

如果 `candidate <= window_end`，它就是下一次提醒；否则切到下一个符合日期过滤条件的日期，再取该日期窗口的第一个点。

比如：

```text
09:00, 09:25, 09:50, 10:15, 10:40 ...
```

当前 10:13，则下一次是 10:15。

### 多窗口处理

一天里可能有多个窗口：

```text
09:00~12:00 every 30m
14:00~18:00 every 20m
22:00 fixed at 22:30
```

算法是：

```text
for each candidate day:
  for each time_window:
    计算该窗口内 >= now 的最早 candidate
  返回所有 candidate 中最小的那个
```

## 7. 时间、时区、休眠恢复策略

这里要提前设计，否则后期 bug 很多。

### 推荐策略

所有存储中的时间规则保留“用户语义时间”，例如：

```text
每周一 09:00
每天 09:00~18:00 每 30 分钟
```

计算触发时转换成 UTC 时间点，交给系统定时器等待。`SetWaitableTimer` 的绝对时间使用 UTC 格式，文档也提醒应使用 UTC-based absolute time。([微软学习](https://learn.microsoft.com/zh-tw/windows/win32/api/synchapi/nf-synchapi-setwaitabletimer?utm_source=chatgpt.com))

### 系统休眠后的处理

电脑睡眠期间错过的提醒，要由 `missed_policy` 决定：

```text
skip              直接跳过错过的提醒
fire_once         恢复后只补一条
fire_all_limited  最多补 N 条，防止刷屏
```

建议默认 `fire_once`，否则用户从睡眠恢复后可能被几十条飘字轰炸。

### 系统时间变更

系统时间、时区、夏令时变化后，应重新计算所有 `next_fire_at`。不应完全依赖旧堆里的时间。

## 8. 飘字显示算法

提醒触发后进入渲染队列：

```text
DisplayEvent
 ├─ reminder_id
 ├─ text
 ├─ priority
 ├─ created_at
 ├─ duration
 ├─ lane
 └─ animation_state
```

滚动动画：

```text
x(t) = screen_width - speed_px_per_sec * elapsed
y = lane_y
```

当 `x + text_width < 0`，说明文字完全飘出屏幕，动画结束。

### 多条提醒同时出现

需要“泳道”机制，避免文字重叠：

```text
lane 0: 顶部第 1 行
lane 1: 顶部第 2 行
lane 2: 顶部第 3 行
```

分配逻辑：

```text
1. 找一个当前没有被占用的 lane
2. 如果都占用：
   - 高优先级插队
   - 普通提醒排队
   - 超过队列长度则合并为“你有 3 条提醒”
```

## 9. 资源占用控制策略

为了实现“极低资源占用”，关键不是 Rust 本身，而是整体行为：

1. **空闲时不渲染、不轮询、不频繁 wake。**
2. **只维护一个全局 WaitableTimer。**
3. **悬浮窗口平时隐藏或不创建。**
4. **动画只在提醒期间运行。**
5. **提醒文字滚动期间限制帧率，例如 30 FPS；非游戏场景不需要 60/144 FPS。**
6. **配置变更才写数据库，不做频繁落盘。**
7. **不用 Chromium/WebView 常驻。**
8. **设置界面可以独立进程，用完即退；Agent 常驻但极小。**

## 11. 最推荐的总体方案

我建议第一版采用：

```text
Rust + windows crate
Win32 原生后台 Agent
Task Scheduler 登录启动
SQLite 本地存储
MinHeap + 单 WaitableTimer 调度
Win32 Layered Topmost Window
Direct2D/DirectWrite 渲染滚动文字
```

这套方案最符合你的目标：**启动快、依赖少、空闲 CPU 接近 0、内存占用低、提醒及时、UI 能真正显示在当前用户桌面上**。
